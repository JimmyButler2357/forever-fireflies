# Core Memories — Database Schema (v1)

Complete database schema for the Core Memories voice-first parenting journal. This document covers every table, column, index, RLS policy, and storage bucket needed from MVP through V3+.

**How to use this document:** During Phase 3 (Supabase Backend), open this doc and write SQL migrations from it. Every table is listed with exact column types and constraints. Pre-wired columns (for future versions) are marked with bold version tags — create them as nullable columns during MVP so the schema won't need major rewrites later.

## 1. Schema Overview

### Entity Relationships

The core data model centers on a **family** — the shared unit that owns children. This is the key architectural decision: even in MVP (where each family has exactly one parent), the family layer exists so that V2 partner sharing is just an INSERT, not a rewrite.

```
profiles (your account)
   └── family_members (your role: owner / partner / contributor)
         └── families (the shared unit — like a shared photo album)
               └── family_children (junction)
                     └── children

entries (memories — created by a profile, tagged to children)
   └── entry_children (junction: which children this entry is about)
   └── entry_tags (junction: topic tags on this entry)
```

Think of it this way: a `family` is like a shared Google Drive folder. Right now only you have access. Later, you can invite your partner and they get the same view — no files need to move, you just add them to the folder.

## 2. Conventions & Standards

### Naming
- **Tables:** `snake_case`, singular (`entry` not `entries` — *exception: we use `entries` because "entry" conflicts with reserved words in some ORMs*)
- **Columns:** `snake_case` — `created_at` not `createdAt`
- **Junction tables:** `{table_a}_{table_b}` — `entry_children`, `entry_tags`
- **Foreign keys:** `{referenced_table}_id` — `family_id`, `profile_id`

### Common Column Patterns

Every table includes these columns unless noted otherwise:

| Column | Type | Purpose |
|--------|------|---------|
| `id` | `uuid` | Primary key, `DEFAULT gen_random_uuid()` |
| `created_at` | `timestamptz` | Row creation time, `DEFAULT now()` |
| `updated_at` | `timestamptz` | Last modification, `DEFAULT now()`, updated via trigger |

### UUID Strategy
- All primary keys are UUIDs generated by PostgreSQL's `gen_random_uuid()`
- Supabase Auth provides `auth.uid()` for the current user's UUID
- No sequential IDs — prevents enumeration attacks and simplifies multi-tenant RLS

### Timestamps
- Always `timestamptz` (timestamp with time zone), stored as UTC
- Displayed in the user's local timezone by the frontend
- `updated_at` auto-maintained by a shared trigger function (see Section 11)

### Soft Delete Pattern
Tables that support soft delete include:
- `is_deleted` (`boolean`, `DEFAULT false`) — marks the row as deleted
- `deleted_at` (`timestamptz`, nullable) — when the deletion happened

Soft-deleted rows are hidden from normal queries via RLS or application logic. After 30 days, a scheduled job hard-deletes them.

### Pre-Wired Column Convention
Columns needed for future versions are included in their parent table's definition but:
- Marked with a **bold version tag** (e.g., **V1.5**, **V2**) in the Version column
- Always `nullable` with `DEFAULT null`
- Explained in a "Pre-wired fields" callout below the table

This avoids painful `ALTER TABLE` migrations on tables that may have millions of rows.

### RLS Policy Naming
- Format: `{table}_{action}_{scope}` — e.g., `entries_select_family`, `children_insert_owner`
- Actions: `select`, `insert`, `update`, `delete`
- Scopes: `own` (user's own rows), `family` (any family member), `owner` (family owner only)

### 3.6 entries

> **Version:** MVP | **RLS:** Yes | **Refs:** FR-004 through FR-008, FR-018, Product Spec §2, §8

**Purpose:** The core table. Every voice recording or text entry creates one row here. This is the table you'll query most — the home screen timeline, Core Memories, and search all read from it.

| Column | Type | Nullable | Default | Version | Notes |
|--------|------|----------|---------|---------|-------|
| `id` | uuid | NO | gen_random_uuid() | MVP | PK |
| `user_id` | uuid | NO | auth.uid() | MVP | FK → profiles.id. Who recorded this entry |
| `family_id` | uuid | NO | — | MVP | FK → families.id. Denormalized for RLS performance |
| `transcript` | text | YES | null | MVP | Editable transcript text. FR-008 |
| `original_transcript` | text | YES | null | MVP | Preserved original before edits. FR-008 |
| `entry_type` | text | NO | 'voice' | MVP | 'voice' or 'text' |
| `entry_date` | date | NO | CURRENT_DATE | MVP | User-selectable for backdating. FR-009b |
| `audio_storage_path` | text | YES | null | MVP | Path in Supabase Storage. Null for text entries |
| `audio_duration_seconds` | smallint | YES | null | MVP | Recording length. Null for text entries |
| `is_favorited` | boolean | NO | false | MVP | Core Memory toggle. FR-018 |
| `is_deleted` | boolean | NO | false | MVP | Soft delete flag |
| `deleted_at` | timestamptz | YES | null | MVP | 30-day recovery window start |
| `mood` | text | YES | null | **V1.5** | Pre-wired. Quick-react mood tag |
| `unlock_at_date` | timestamptz | YES | null | **V3+** | Pre-wired. Time Capsule: unlock on this date |
| `unlock_at_age_months` | smallint | YES | null | **V3+** | Pre-wired. Time Capsule: unlock when child reaches this age |
| `created_at` | timestamptz | NO | now() | MVP | |
| `updated_at` | timestamptz | NO | now() | MVP | |

**Pre-wired fields:**
- `mood` (V1.5) — quick-react mood/emotion tag (laughing, crying, proud, exhausted, grateful). Added as a column rather than a separate table for simplicity since it's always 1:1 with the entry. *(See Product Spec §7, V1.5 roadmap)*
- `unlock_at_date` (V3+) — Time Capsule feature. A message that "unlocks" on a specific date (e.g., "Open on your 18th birthday"). Null means the entry is visible immediately. *(See Product Spec §7, Brainstorm Parking Lot)*
- `unlock_at_age_months` (V3+) — Time Capsule by age. Unlocks when the tagged child reaches a certain age in months (e.g., 216 = 18 years). Alternative to a fixed date — the app calculates unlock date from the child's birthday.

**Why `family_id` is denormalized here:** RLS policies need to check "does this user belong to the family that owns this entry?" On every query. Joining through `profiles → family_members → families` on every row is expensive. Storing `family_id` directly on entries lets RLS do a simple subquery: `family_id IN (SELECT family_id FROM family_members WHERE profile_id = auth.uid())`.

**Notes:**
- `user_id` is the "recorded_by" field — it tells you which parent created this entry. This is the field that makes V2 partner sharing work: both parents create entries, and each entry shows who recorded it.
- `audio_storage_path` follows the pattern: `{user_id}/{entry_id}.wav` (MVP) or `.m4a` (post-optimization)
- Soft-deleted entries are hidden via RLS or app logic, then hard-deleted after 30 days by a scheduled Edge Function

**RLS policies:**
- `entries_select_family` — users can read entries belonging to their family
- `entries_insert_own` — users can create entries in their own family
- `entries_update_own` — users can edit their own entries (transcript, tags, favorite, date)
- `entries_update_family_favorite` — family members can toggle favorite on any family entry
- `entries_delete_own` — users can soft-delete their own entries

## 7. What Doesn't Live in the Database

Not everything needs a table. These services manage their own data:

### RevenueCat (Subscriptions & Billing)
RevenueCat is the **source of truth** for all subscription state. It manages trial tracking, payment processing, receipt validation, and subscription lifecycle. The `profiles.subscription_status` column is a convenience cache — synced from RevenueCat via webhook → Edge Function. Never query your own database to determine if a user can access premium features; query RevenueCat's SDK instead.

### PostHog (Analytics)
All analytics events (entry_recorded, onboarding_completed, subscription_started, etc.) go directly to PostHog via their SDK. No local `events` table is needed. PostHog handles retention analysis, funnel tracking, and cohort analysis.

### Supabase Auth (`auth.users`)
Supabase manages the `auth.users` table automatically. You never write to it directly. The `profiles` table extends it with app-specific fields via a trigger. Authentication tokens, sessions, and social provider data are all handled by Supabase Auth.

## 10. Row Level Security (RLS) Policies

> **Every table gets RLS enabled before the first row is inserted.** This is non-negotiable — it's trivial during table creation and painful to retrofit.

### Core RLS Pattern

Most policies follow the "family membership" pattern. Think of it as: "You can see something if you're in the same family as it."

```sql
-- Helper: Get the current user's family IDs
-- Used in most RLS policies
CREATE FUNCTION user_family_ids()
RETURNS SETOF uuid AS $$
  SELECT family_id
  FROM family_members
  WHERE profile_id = auth.uid()
    AND status = 'active'
$$ LANGUAGE sql SECURITY DEFINER STABLE;
```

### Policy Summary

| Table | SELECT | INSERT | UPDATE | DELETE |
|-------|--------|--------|--------|--------|
| `profiles` | Own only | Auto (trigger) | Own only | — |
| `families` | Family member | Auto (trigger) | Owner only | — |
| `family_members` | Family member | Owner only | Own row | Owner only |
| `children` | Family member | Family member | Family member | Owner only |
| `family_children` | Family member | Family member | — | Owner only |
| `entries` | Family member | Own family | Own entries | Own entries |
| `entry_children` | Family member | Own entries | Own entries | Own entries |
| `tags` | All (system) + family (custom) | Family member | Family member | Family member |
| `entry_tags` | Family member | Own entries | Own entries | Own entries |
| `prompts` | All authenticated | — | — | — |
| `user_devices` | Own only | Own only | Own only | Own only |
| `prompt_history` | Own only | Own only | — | — |
| `notification_log` | Own only | Service role only | — | — |

**"Owner only"** means the user must have `role = 'owner'` in `family_members` for that family. This prevents partners from deleting children or revoking each other's access.
